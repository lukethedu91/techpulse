Docker for Beginners: What It Is and Why Developers Love It

Containers are the reason modern software can be deployed in minutes and run identically everywhere. If you've heard of Docker but never quite got it, this is the guide.

By Staff Writer  ·  March 2026  ·  8 min read

At some point every developer has said "it works on my machine" with that specific mix of frustration and defensiveness. It works on your machine because your machine has the exact version of Node you installed eight months ago, the exact system library version that your dependency subtly requires, the exact environment variable set in your profile that you forgot about. None of that is on the server.

Docker's core insight is that the "machine" context — the OS libraries, the runtime versions, the filesystem structure your code expects — should be packaged with the code, not assumed from the environment. A container is that package: your application plus everything it needs to run, in a portable unit that runs identically on your laptop, a CI server, and a production machine.

The Core Concepts

Three things to understand:

Image: A read-only template. An image for a Python web app might contain Ubuntu, Python 3.12, your pip dependencies, and your code. Images are built from a Dockerfile and can be stored in a registry (Docker Hub, GitHub Container Registry, etc.).

Container: A running instance of an image. You can run multiple containers from the same image. Each container is isolated — it has its own filesystem, its own network interface, its own process space.

Dockerfile: A text file with instructions for building an image. It starts from a base image (usually a minimal OS or language runtime) and adds layers — install dependencies, copy code, set the startup command.

A simple Python app Dockerfile:

FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]

Build it with docker build -t my-app . and run it with docker run -p 8000:8000 my-app. The -p flag maps port 8000 on your machine to port 8000 inside the container.

Why This Matters in Practice

The obvious use case is deployment consistency. Build the image once in CI, push it to a registry, pull and run it on any server. The container is identical to what passed tests.

The less obvious use case is local development environments. Instead of installing MySQL, Redis, and a specific version of Node globally on your machine (and then having conflicts when another project needs different versions), you define them in a docker-compose.yml and spin them up for each project independently:

services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: localdev
  redis:
    image: redis:7-alpine
  app:
    build: .
    ports:
      - "3000:3000"
    depends_on:
      - db
      - redis

Run docker compose up and you have a fully running local stack. Run docker compose down and it disappears cleanly. Switch to a different project with different versions of everything — same workflow, no conflicts.

The Overhead Question

Containers are not virtual machines. A VM emulates hardware and runs a full OS — heavy, slow to start, meaningful memory overhead. A container shares the host OS kernel and only adds the application's dependencies on top. Starting a container typically takes under a second. Memory overhead is the actual memory your application uses, not a separate OS instance.

On Linux, containers run natively. On Mac and Windows, Docker Desktop runs a lightweight Linux VM in the background — there's a small overhead, but nothing like running a full VM for each container.

"The container doesn't care that your staging server has a different Node version than prod. It brought its own."

Docker Hub and Images

Docker Hub (hub.docker.com) hosts public images for almost every piece of software you'd want to run: nginx, postgres, redis, mysql, node, python, official images for every major database and language runtime. You can use these as base images in your Dockerfile or run them directly.

Running a temporary PostgreSQL database for local testing:

docker run --rm -e POSTGRES_PASSWORD=test -p 5432:5432 postgres:16

The --rm flag removes the container when you stop it. Useful for throwaway test environments you don't want lingering around.

What to Learn Next

Once you're comfortable with basic container management: Docker Compose for multi-container setups (covered above), then volumes for persistent data (so your database survives container restarts), then Docker networking for more complex service communication. Kubernetes is the orchestration layer beyond that, but it's a significant complexity jump — most small-to-medium projects don't need it and are better served by Docker Compose or a simple managed container service.
